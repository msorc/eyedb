EyeDB comparative study specifications

Goals of the study

The goals are the following:
  • have precise performance measurement
  • compare EyeDB with other object persistency solutions, in term of performance but also functionalities,
  • improve EyeDB performance in order to be faster than other solutions

What are the competitors to evaluate

Java:

  • Hibernate
  • db4o
  • JDBC
  • JDO
  • other proprietary solutions: versant?

C++:

  • to be completed

Benchmark framework

The benchmarks will address only the object persistence layer, without any integration in a web application server like Tomcat or other:

  • no need to address the problem of database connections pooling
  • no problem of concurrency

However, as integrating EyeDB in a web application environment is an interesting target and requires some new developments, this point should be addressed later.

The benchmarks will therefore consist in comparing execution time, memory, machine load (other points?) when running objects persistency operations:

  • object creation
  • object loading
  • object querying (is there a difference between loading and querying? in Hibernate yes, you can load an object without using the query language)
  • object modification
  • object deletion

Benchmark application

In order to have meaningful benchmarks, a real application and data model must be chosen, that contains: inheritance, relations (all cardinality), what else?

Instead of reusing the "quicktour", we will reuse the "petshop" (or petstore) application, a classical example application in the Java world that implements an on-line shop. Several sources of the petshop exist:

  • https://betterpetshop.dev.java.net/
  • http://java.sun.com/developer/releases/petstore/
  • http://java.sys-con.com/node/46977

The benchmark application will reuse from the petshop application only the data model, with possible additions in order to use all EyeDB features (in particular relationship and relational integrity). An ODL model of the petshop is already available (see in this directory the ODL file).

The data model will be implemented by a DAO [http://en.wikipedia.org/wiki/Data_Access_Object] engine, in order to separate as much as possible the benchmark code from the implementation of the access to the persistence layer. This pattern is used widely in the Spring framework [http://www.springframework.org], but a priori the benchmark will not use the Spring framework (to be further evaluated).
